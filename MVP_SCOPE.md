# MVP_SCOPE — ИИ-ассистент по внутренним заявкам на закупку

**Проект:** ИИ-ассистент по внутренним заявкам на закупку  
**Версия:** MVP 1.0  
**Цель документа:** зафиксировать границы первой рабочей версии и не допустить расширения объёма разработки до завершения базового сценария.

## 1. Цель MVP

Создать работающего Telegram-ассистента, который помогает внутреннему заказчику оформить нерегулярную заявку на закупку товара или услуги, отвечает на вопросы по базе знаний и сохраняет подтверждённую заявку в системе.

MVP должен доказать, что ассистент способен:

- понимать запрос пользователя на естественном языке;
- использовать корпоративную базу знаний через RAG;
- последовательно собирать недостающие сведения;
- формировать структурированную заявку;
- сохранять черновик и подтверждённую заявку;
- корректно обрабатывать отсутствие информации и технические ошибки.

## 2. Основной пользовательский сценарий

1. Пользователь пишет в Telegram, что ему нужно приобрести товар или заказать услугу.
2. Ассистент определяет намерение пользователя.
3. Ассистент определяет тип закупки: товар или услуга.
4. Ассистент предлагает подходящую категорию закупки.
5. Ассистент извлекает уже указанные данные.
6. Ассистент получает обязательные поля для выбранной категории.
7. Ассистент последовательно задаёт вопросы только по недостающим или противоречивым данным.
8. После заполнения обязательных полей ассистент формирует итоговую карточку заявки.
9. Пользователь подтверждает передачу заявки.
10. Система присваивает номер, сохраняет заявку и устанавливает внутренний код статуса `new`.
11. Пользователь получает подтверждение регистрации.

Пользователю внутренний статус `new` отображается как «Передана в отдел закупок».

## 3. Функции, входящие в MVP

### 3.1. Работа с заявками

- создание новой заявки;
- определение типа: товар или услуга;
- предварительная классификация закупочной категории;
- извлечение данных из свободного текста;
- проверка обязательных полей;
- последовательные уточняющие вопросы;
- формальная проверка количества, единиц измерения, дат, бюджета и логических противоречий;
- обработка конкретного бренда и запрета эквивалента;
- обработка срочной заявки;
- формирование итоговой карточки;
- явное подтверждение пользователем;
- присвоение уникального номера;
- сохранение заявки с внутренним кодом статуса `new`, который отображается пользователю как «Передана в отдел закупок».

### 3.2. Черновики

- автоматическое сохранение промежуточных данных;
- продолжение последнего незавершённого черновика;
- сохранение данных при временной технической ошибке;
- отсутствие регистрации без явного подтверждения пользователя.

### 3.3. Ответы по базе знаний

- ответы на вопросы по правилам подачи и обработки заявок;
- поиск релевантных фрагментов через RAG;
- указание названия документа-источника;
- краткое практическое действие для пользователя;
- fallback, если подтверждённого ответа в базе нет.

### 3.4. Telegram-интерфейс

Минимальные команды:

- `/start`
- `/new`
- `/continue`
- `/cancel`
- `/help`

Основные действия:

- создать заявку;
- продолжить черновик;
- задать вопрос по закупочному процессу;
- получить помощь.

### 3.5. Логирование

Для каждого обращения сохраняются:

- пользователь;
- запрос;
- ответ;
- намерение;
- модель;
- использованные источники;
- количество токенов;
- длительность обработки;
- ошибка, если она возникла.

## 4. Функции, не входящие в MVP

В первой версии не реализуются:

- автоматическое согласование заявок;
- фактическая маршрутизация по согласующим;
- уведомления руководителей и финансовых контролёров;
- полноценный кабинет закупщика;
- ручное изменение статусов закупщиком;
- автоматическая проверка бюджета;
- выбор или рекомендация поставщика;
- тендер, RFQ и сбор коммерческих предложений;
- создание заказа поставщику;
- договорная работа;
- оплата;
- интеграции с ERP, 1С, CRM и WMS;
- загрузка и анализ вложений;
- каталог товаров и услуг;
- проверка складских остатков;
- аналитические дашборды;
- голосовой ввод;
- автоматическое выявление дублей;
- сложная ролевая модель и SSO;
- мобильное приложение;
- reranker;
- n8n-сценарии, не необходимые для основного процесса.

Эти функции могут быть добавлены после успешного завершения и тестирования базового сценария.

## 5. Роли компонентов

### LLM

- определяет намерение;
- извлекает поля из сообщения;
- предлагает категорию;
- выявляет возможные противоречия;
- формулирует уточняющий вопрос;
- создаёт понятный пользователю текст ответа и итоговой карточки.

### Код приложения

- хранит состояние диалога;
- получает обязательные поля;
- выбирает следующее незаполненное поле;
- выполняет формальные проверки;
- управляет черновиками;
- запрещает регистрацию без подтверждения;
- присваивает номер заявки;
- сохраняет данные;
- управляет статусом.
- детерминированно рассчитывает предварительный маршрут по структурированным
  правилам, не выполняя само согласование.

### RAG

- ищет правила и объяснения в базе знаний;
- передаёт в модель только релевантные фрагменты;
- возвращает название документа и раздела.

### Supabase

- хранит пользователей;
- хранит заявки и черновики;
- хранит состояние диалога;
- хранит чанки и embeddings;
- хранит технические логи.

## 6. Ограничения MVP

- используются только синтетические данные;
- интерфейс — Telegram;
- язык — русский;
- пилотная аудитория — 5–10 внутренних заказчиков и 2 сотрудника закупок;
- база знаний — 13–15 Markdown-документов;
- основная нагрузка — до 30 зарегистрированных заявок и до 500 сообщений в день;
- время ответа — менее 30 секунд;
- ассистент не принимает управленческие, финансовые, юридические и закупочные решения;
- ассистент не использует общие знания модели как источник внутренних правил компании;
- при отсутствии ответа ассистент направляет пользователя к сотруднику отдела закупок.

## 7. Критерии готовности MVP

MVP считается готовым, если:

1. Telegram-бот стабильно принимает сообщения.
2. Ассистент различает создание заявки и справочный вопрос.
3. Ответы по базе знаний содержат корректный источник.
4. Fallback срабатывает без выдуманных правил.
5. Ассистент способен пройти полный диалог по товарной заявке.
6. Ассистент способен пройти полный диалог по заявке на услугу.
7. Ассистент не задаёт повторно вопросы по уже заполненным полям.
8. Обязательные поля определяются по категории.
9. Итоговая карточка содержит все собранные данные.
10. Заявка не регистрируется без явного подтверждения.
11. После подтверждения заявка получает уникальный номер, внутренний код статуса `new` и отображаемое пользователю название «Передана в отдел закупок».
12. Черновик сохраняется и восстанавливается.
13. При технической ошибке система не сообщает об успешной регистрации.
14. Среднее время ответа не превышает 30 секунд.
15. Подготовленный тестовый набор проходит без критических ошибок.

## 8. Минимальный тестовый набор

- 5 типовых вопросов по базе знаний;
- 2 вопроса вне базы;
- 2 follow-up вопроса;
- 1 провокационная попытка изменить роль ассистента;
- 1 простая товарная заявка;
- 1 заявка на услугу;
- 1 неполная заявка;
- 1 противоречивая заявка;
- 1 заявка с конкретным брендом;
- 1 срочная заявка;
- 1 запрос с несколькими закупочными категориями;
- 1 сохранение и восстановление черновика;
- 1 сценарий технической ошибки.

## 10. Правило изменения границ

Новая функция добавляется в MVP только если одновременно выполняются три условия:

1. без неё невозможно завершить основной пользовательский сценарий;
2. она не создаёт заметный риск для срока и стабильности;
3. она согласована до начала реализации соответствующего этапа.

Все остальные идеи фиксируются в backlog и не реализуются до завершения тестирования MVP 1.0.
