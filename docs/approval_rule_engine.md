# Движок правил согласования

## Назначение

Retrieval находит нормативный источник, но не должен вычислять диапазоны сумм
или применять бизнес-условия. `ApprovalRuleService` детерминированно
рассчитывает предварительный маршрут по структурированному контексту.

Движок не извлекает данные из свободного текста, не генерирует ответ, не
согласовывает заявку и не изменяет её статус.

## Входной контекст

`ApprovalContext` содержит сумму как Decimal, бюджетный статус, срочность,
признак единственного поставщика, category code, доступ к данным, работы на
объекте и необязательную дату оценки. Категория нормализуется в верхний
регистр. Для юридических услуг используется существующий код `S11`.

## Базовые правила

| Код правила | Статус бюджета | Сумма | Согласующие |
|---|---|---:|---|
| `BUDGETED_0_100000` | budgeted | 0.00–100000.00 | Руководитель подразделения |
| `BUDGETED_100000_01_500000` | budgeted | 100000.01–500000.00 | Руководитель подразделения; Финансовый контролёр |
| `BUDGETED_500000_01_PLUS` | budgeted | от 500000.01 | Руководитель подразделения; Финансовый блок; Руководитель закупок |
| `UNBUDGETED_0_100000` | unbudgeted | 0.00–100000.00 | Руководитель подразделения; Финансовый директор |
| `UNBUDGETED_100000_01_PLUS` | unbudgeted | от 100000.01 | Руководитель подразделения; Финансовый директор; Генеральный директор |

Границы включительны и соответствуют `kb-009 > Матрица согласования`.

## Дополнительные правила

- P1/P2 добавляет Руководителя закупок;
- единственный поставщик добавляет Руководителя закупок;
- единственный поставщик свыше 500000 добавляет Финансовый блок;
- доступ к данным добавляет IT / информационную безопасность;
- категория S11 добавляет Юридическую службу;
- работы на объекте добавляют АХО / охрану труда.

Правила применяются по `priority`, затем `rule_code`. Дубли согласующих
удаляются с сохранением первого положения.

## Статусы результата

- `resolved` — найдено ровно одно базовое правило;
- `needs_clarification` — отсутствует бюджетный статус;
- `no_matching_rule` — активного правила нет;
- `conflict` — подошло несколько базовых правил.

При конфликте правило не выбирается молча: возвращаются предупреждение и
ссылки на источники.

## Хранилище и обновление

Человекочитаемый seed: `data/rules/approval_rules.json`. Миграция
`scripts/sql/005_approval_rule_engine.sql` создаёт `approval_base_rules` и
`approval_additional_rules`, затем идемпотентно обновляет seed через
`ON CONFLICT (rule_code) DO UPDATE`.

Для уже развёрнутой старой версии 005 миграция
`scripts/sql/006_fix_approval_rule_ranges.sql` заменяет прежние коды
правил и исправляет денежные границы с точностью до 0.01. Она безопасна
и при чистом развёртывании с уже исправленной 005: существующие новые
коды только получают те же значения повторно.

Проверка перед обновлением:

```powershell
python scripts/validate_approval_rules.py
```

Валидатор проверяет коды, диапазоны, даты, согласующих, источники, разделы и
category code.

## API и CLI

```http
POST /api/v1/approval-rules/evaluate
GET  /api/v1/approval-rules/health
```

```powershell
python scripts/evaluate_approval_route.py `
  --amount 180000 `
  --budget-status budgeted
```

API только рассчитывает маршрут. Он не принимает секреты и не пишет в заявки.

## Ограничения MVP

- параметры должны быть уже структурированы;
- нет LLM extraction и универсального DSL;
- нет фактической маршрутизации, уведомлений или согласования;
- нет автоматической проверки бюджета;
- изменения правил требуют проверки владельцем.
