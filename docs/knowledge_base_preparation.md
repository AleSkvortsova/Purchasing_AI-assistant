# Подготовка базы знаний

## Структура

В `knowledge_base/` находятся 15 Markdown-файлов: служебный
`00_README.md` и 14 индексируемых источников `01_…md`–`14_…md`.
Loader намеренно исключает `00_README.md`; он описывает набор документов, но
не используется как источник ответа. Проектные документы остаются в `docs/`.

Автоматически созданный `knowledge_base/manifest.json` содержит сведения о
14 индексируемых источниках и их SHA-256.

## YAML front matter

Каждый Markdown-файл содержит:

```yaml
---
document_id: kb-001
title: Регламент подачи и обработки заявок
document_type: regulation
version: "1.0"
effective_date: "2026-08-01"
owner: Отдел закупок
priority: 1
status: active
language: ru
---
```

`document_id` должен быть уникальным, дата — иметь формат `YYYY-MM-DD`,
а `owner` — быть заполнен. Поддерживаются типы:

- `regulation`;
- `template`;
- `classifier`;
- `field_matrix`;
- `urgency_rules`;
- `status_guide`;
- `responsibility_matrix`;
- `approval_rules`;
- `faq`;
- `examples`;
- `glossary`;
- `user_guide`;
- `error_guide`;
- `overview`.

## Приоритеты источников

1. Регламент.
2. Матрица обязательных полей.
3. Классификатор категорий.
4. Статусы и переходы.
5. Правила согласования.
6. Правила срочности.
7. FAQ.
8. Примеры.
9. Шаблоны и остальные поясняющие документы.

При конфликте ответа приоритет имеет источник с меньшим числом `priority`.
Смысловой конфликт всё равно требует ручной проверки владельцем базы знаний.

## Чанкинг

Документ сначала делится по Markdown-заголовкам уровней `#`–`####`.
Каждый чанк сохраняет полный `section_path`. Заголовок остаётся вместе с
текстом, а в начало `content` добавляются название документа и путь раздела.

Ориентир размера — 400–600 токенов, мягкий максимум — 700. Пока используется
приближение:

```text
ceil(количество символов / 3)
```

Длинный раздел дополнительно делится по абзацам, элементам списков, затем по
предложениям. Между частями используется overlap около 60 токенов.
Markdown-таблицы, fenced code blocks и связанные блоки «Плохо / Хорошо»
считаются атомарными и не разрезаются. Поэтому отдельный атомарный блок может
осознанно превысить мягкий максимум.

## Стабильный `chunk_id`

ID строится через UUID5 из:

- `document_id`;
- `section_path`;
- нормализованного текста конкретной части.

Повторный запуск с неизменными входными файлами даёт те же ID. Изменение
содержания меняет ID затронутого чанка.

## Запуск

Из корня проекта:

```powershell
python scripts/prepare_knowledge_base.py
```

Скрипт загружает документы, валидирует их, создаёт manifest и чанки, затем
сохраняет:

- `data/processed/knowledge_documents.json`;
- `data/processed/knowledge_chunks.json`;
- `data/processed/validation_report.json`;
- `data/processed/chunk_statistics.json`.

JSON записывается в UTF-8 с `ensure_ascii=False` и стабильной сортировкой.
При наличии ошибок скрипт завершается с ненулевым кодом и не публикует
documents/chunks/manifest как финальный результат. Предупреждения обработку не
останавливают.

## Validation report

Ошибками считаются, среди прочего, отсутствующий H1, неизвестный тип, неверная
дата, пустой owner и повторяющийся `document_id`. Предупреждения включают
длинные разделы, повторяющиеся заголовки, возможные дубли, неактивный статус и
известные терминологические расхождения.

## Добавление и обновление

1. Добавить или изменить Markdown в `knowledge_base/`.
2. Заполнить front matter и назначить уникальный `document_id`.
3. Не изменять приоритет без согласования с владельцем.
4. Запустить подготовку.
5. Проверить validation report, статистику и крупнейшие чанки.
6. Просмотреть diff manifest и processed JSON.

После будущего подключения embeddings любое содержательное изменение источника
потребует повторной подготовки чанков и переиндексации изменившихся
`chunk_id`. На текущем этапе embeddings, векторное хранилище и RAG не
используются.
